基于美团Robust客户端补丁流程方案设计

Posted on 2018-05-27 | In Android

基于美团Robust客户端补丁流程方案设计

首先来看一下Robust类中的一些初始化定义

  • 从RobustParamsProvider中取出一些参数如：userId,uuid,channel,version等信息
  • 最终调用PatchUtils.applyPatch(context, "duitang", versionName, channel, userID, uuid, PatchUtils.USE_NONE, robustCallBack);

PatchUtils

  • 拿到当前app的hash值
  • PatchManipulateImpl是补丁获取和应用补丁具体方法的实现类(这是客户端上层实现补丁获取的方式以及应用所暴露出的接口)
  • 最后传入PatchManipulateImpl对象到PatchHelper中去执行

PatchExecutor

PatchExecutor是一个Thread，我们来具体看一下run方法

  • 可以看到 patchManipulate.fetchPatchList就是实现了对外暴露获取补丁方式的接口public abstract class PatchManipulate extends Object的实现类 ，去调用实现这个类的获取补丁的方法,即patchManipulate.fetchPatchList

下面来一下PatchManipulateImpl对象最核心的实现方法

  • 请求补丁列表的json地址(通常是从server端的某一个接口获取)
  • 通过Gson将返回的补丁json信息解析成补丁对象Patch
  • tmp.setLocalPath(getPatchDirPath(context) + p.getMd5() + "_" + this.apkHash);这里主要是设置下载补丁到本地路径的localPath
  • updateLocalPatchListDelay最终会调用 PatchHelper.updatePatchListJsonToLocal updatePatchListJsonToLocal主要是将本地跟此版本对应的补丁对象与server返回的Patch对象进行一下比较，如果本地的Patch对象跟返回的server补丁对象不一致，就删除本地的Patch对象
  • 很明显setInfoAndMergePatches方法是将server返回的补丁对象补充一些信息,如LocalPath,appHash,setPatchesInfoImplClassFullName这个属性是设置补丁的入口类，默认是com.meituan.robust.patch.PatchesInfoImpl

好了获取补丁的流程到这已经差不多了，下面是应用补丁的过程，此过程是在Robust框架下完成的(即在Robust依赖库中完成)

  • resourcesPatches这个对象说明Robust框架已经可以支持资源修复了(日后会研究)
  • 这个方法很重要，通过补丁的localPath判断本地是否下载过，如果有就不会再下载，避免了获取补丁信息之后重复下载,如果本地不存在则从网络下载真正的补丁包

  • 通过比较md5验证下载下来补丁的合法性(为了防止无论是本地存在的补丁包还是从cdn下载的补丁包被篡改)

  • int patchType = PatchTypeUtil.getPatchType(p);获取补丁包的类型，如果是class.dex表示只是代码修复补丁

下面就是最核心的补丁修复原理的代码了

  • 可以看到修复原理主要利用系统的DexClassLoader来进行loadClass,所有确实Robust在热修复的兼容性上是能足够保证的
  • 将实现了ChangeQuickRedirectField接口的补丁逻辑类设置给之前预埋补丁代码的类的changeQuickRedirectField属性中，预埋代码的部分是在打包过程中完成的，修复完成后用户除了无感知之外并且也不需要重启app，直接无痕修复了线上的bug

总结

上述逻辑主要是根据Robust框架的修复流程来设计的客户端应用此框架的一个流程，主要从获取补丁的稳定性以及保护补丁的安全性入手，加入了一些安全读写补丁文件工具类，后续会对有专门的文章来分析它们的实现
